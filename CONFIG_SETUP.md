# RepoWise Stable Server Configuration

This guide explains how to configure RepoWise for a stable server setup using environment files and config files, without relying on CLI switches.

## Directory Structure

```
your-repo/
├── .repowise/
│   ├── .env                 # Environment variables (local, not versioned)
│   ├── config.yaml          # Repository configuration (auto-generated, can be versioned)
│   ├── wiki.db              # Database (local)
│   └── lancedb/             # Vector store (local)
├── .gitignore
└── other-files...
```

## Step 1: Create `.repowise/.env`

This file stores connection details and API keys. **Never commit this to git.**

```bash
# .repowise/.env
OPENAI_COMPATIBLE_BASE_URL=http://localhost:11434/v1
# OPENAI_COMPATIBLE_API_KEY=  # Leave empty for local Ollama
```

**For other setups:**

```bash
# LM Studio
OPENAI_COMPATIBLE_BASE_URL=http://localhost:1234/v1

# vLLM with auth
OPENAI_COMPATIBLE_BASE_URL=https://my-vllm.example.com/v1
OPENAI_COMPATIBLE_API_KEY=sk-your-key-here

# Azure OpenAI
OPENAI_COMPATIBLE_BASE_URL=https://my-resource.openai.azure.com/openai/deployments/gpt-4
OPENAI_COMPATIBLE_API_KEY=your-azure-key
```

## Step 2: Create `.repowise/config.yaml`

This file stores provider and model choices. You can commit this to git (without secrets).

```yaml
# .repowise/config.yaml
# RepoWise configuration — auto-generated by init, can be edited manually

# ============================================================================
# LLM Provider Configuration (Chat)
# ============================================================================
provider: openai_compatible
model: qwen2.5-coder:3b

# ============================================================================
# Embedder Configuration (RAG/Vector Search)
# ============================================================================
embedder: openai_compatible
embedder_model: nomic-embed-text-v2-moe:latest

# ============================================================================
# Optional: Generation Settings
# ============================================================================
# reasoning: auto
# exclude_patterns:
#   - tests/
#   - vendor/
# commit_limit: 500
# follow_renames: false
```

**Key fields:**
- `provider`: LLM provider (anthropic, openai, openai_compatible, etc.)
- `model`: Model name for the provider
- `embedder`: Embedding provider (gemini, openai, openai_compatible, openrouter)
- `embedder_model`: Embedding model name (Ollama, Azure, etc.)

## Step 3: Add to `.gitignore`

Ensure credentials and local data are never committed:

```bash
# .gitignore
.repowise/.env
.repowise/wiki.db
.repowise/lancedb/
```

The `config.yaml` *can* be committed (it has no secrets).

## Step 4: Run Commands Without CLI Flags

Once configured, all commands automatically use the saved settings:

```bash
# Initialize (uses config.yaml settings)
repowise init

# Update documentation
repowise update

# Reindex vectors
repowise reindex

# Start the server
repowise serve
```

## How It Works: Resolution Order

### For LLM Provider
1. CLI flag: `--provider openai_compatible --model qwen2.5-coder:3b`
2. Environment: `REPOWISE_PROVIDER` and `REPOWISE_DEFAULT_MODEL` env vars
3. Config file: `provider` and `model` in `.repowise/config.yaml`
4. Auto-detect: From available API keys (OPENAI_API_KEY, ANTHROPIC_API_KEY, etc.)

### For Embedder
1. CLI flag: `--embedder openai_compatible --embedder-model nomic-embed-text-v2-moe:latest`
2. Environment: `REPOWISE_DEFAULT_EMBEDDER` env var
3. Config file: `embedder` and `embedder_model` in `.repowise/config.yaml`
4. Auto-detect: From available API keys

## Example: Complete Setup for Ollama

### Step 1: Start Ollama with both models

```bash
# Terminal 1
ollama serve

# Terminal 2 (pull models)
ollama pull qwen2.5-coder:3b
ollama pull nomic-embed-text-v2-moe:latest
```

### Step 2: Create `.repowise/.env`

```bash
# .repowise/.env
OPENAI_COMPATIBLE_BASE_URL=http://localhost:11434/v1
```

### Step 3: Create `.repowise/config.yaml`

```yaml
# .repowise/config.yaml
provider: openai_compatible
model: qwen2.5-coder:3b
embedder: openai_compatible
embedder_model: nomic-embed-text-v2-moe:latest
```

### Step 4: Initialize and run

```bash
# First time: index the repo
repowise init

# Subsequent times: just run update
repowise update

# Start the server
repowise serve
```

## Example: Complete Setup for Microsoft Foundry

RepoWise works seamlessly with Microsoft Foundry using the `openai_compatible` provider with API key authentication.

### Prerequisites

1. Microsoft Foundry resource created in Azure
2. A model deployment (e.g., `gpt-4o-mini`, `gpt-4o`)
3. API key from your Foundry resource (Keys & Endpoints page in Azure Portal)
4. Resource name (e.g., `my-foundry-resource`)

### Step 1: Create `.repowise/.env`

Store your Foundry credentials here (never commit this):

```bash
# .repowise/.env
# Microsoft Foundry endpoint (replace YOUR-RESOURCE-NAME with your actual resource)
OPENAI_COMPATIBLE_BASE_URL=https://YOUR-RESOURCE-NAME.openai.azure.com/openai/v1
OPENAI_COMPATIBLE_API_KEY=your-foundry-api-key-here
```

**Note:** Find your resource name and API key in:
- Azure Portal → Your Resource → Keys and Endpoint
- Or Microsoft Foundry portal (ai.azure.com)

### Step 2: Create `.repowise/config.yaml`

Use deployed models from your Foundry resource:

```yaml
# .repowise/config.yaml
# Chat model (use your deployed model name)
provider: openai_compatible
model: gpt-4o-mini

# Embeddings model (use your deployed embedding model)
embedder: openai_compatible
embedder_model: text-embedding-3-small
```

**Available models** (depends on your Foundry deployments):
- Chat: `gpt-4o`, `gpt-4o-mini`, `gpt-4-turbo`, `deepseek-v3`, etc.
- Embeddings: `text-embedding-3-small`, `text-embedding-3-large`, etc.

### Step 3: Initialize and run

```bash
# First time: index the repo with Foundry models
repowise init

# Update documentation
repowise update

# Start the server (API will use your Foundry resource)
repowise serve
```

The server will now use Microsoft Foundry for all chat and embedding operations.

### Step 4 (Optional): Use a Centralized Config Directory

If you want to share config across multiple repositories:

```powershell
# Set environment variable to centralized location
$env:REPOWISE_CONFIG_DIR = "C:\tools\repowise"

# Copy .env and config.yaml to that location
Copy-Item .repowise\config.yaml "C:\tools\repowise\config.yaml"
Copy-Item .repowise\.env "C:\tools\repowise\.env"

# Now run commands from any repo — they'll use the centralized config
repowise serve
```

## Switching Models Without CLI

To switch models without using CLI flags, just edit `config.yaml`:

```yaml
# Change the chat model
model: mistral-7b

# Change the embedding model
embedder_model: all-minilm-l6-v2
```

Then run commands as usual:

```bash
repowise reindex  # Rebuilds with new embedding model
repowise init --force  # Regenerates with new chat model
```

## Environment Variables (Optional)

You can also set global defaults via environment:

```bash
# Set globally in your shell or system
export OPENAI_COMPATIBLE_BASE_URL=http://localhost:11434/v1
export REPOWISE_DEFAULT_PROVIDER=openai_compatible
export REPOWISE_DEFAULT_EMBEDDER=openai_compatible
```

These complement the config file — config file takes precedence for repo-specific settings.

## Troubleshooting

**"Provider setup failed; falling back to index-only"**
- Check `.repowise/.env` exists with correct `OPENAI_COMPATIBLE_BASE_URL`
- Verify Ollama/server is running
- Check logs: `repowise init --debug`

**"No real embedder available"**
- Ensure `OPENAI_COMPATIBLE_BASE_URL` is set in `.repowise/.env`
- Run: `repowise reindex --embedder openai_compatible`

**Models not specified**
- Make sure `config.yaml` has both `model` and `embedder_model` fields
- Or use CLI flags to set them once: `repowise init --provider openai_compatible --model qwen2.5-coder:3b --embedder openai_compatible --embedder-model nomic-embed-text-v2-moe:latest`
- This saves to config.yaml for future runs
